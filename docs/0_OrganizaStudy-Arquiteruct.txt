🏗️ Decisões de Arquitetura
Por que services.py por app?
Segue o princípio da Responsabilidade Única (SOLID-S). A view só orquestra HTTP; o model só define estrutura de dados; o service contém toda a regra de negócio — isso torna o código testável de forma unitária sem precisar de um request Django.
Por que UUID como PK?
Elimina IDs sequenciais previsíveis (segurança), facilita merge de dados entre ambientes e prepara a base para uma eventual migração para microsserviços ou API REST.
Por que settings modulares?
Permite que development.py use SQLite com DEBUG=True e production.py use PostgreSQL com ALLOWED_HOSTS e storage em nuvem — sem alterar uma única linha de código da aplicação.
